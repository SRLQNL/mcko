ОБЗОР
-----
Приложение построено на Python как единый фоновый процесс с GUI на tkinter.
Работает без видимых окон при старте. Окно чата создаётся по требованию и
скрывается, а не уничтожается, чтобы сохранить историю сессии.

МОДУЛЬ: InputField (ui/input_field.py)
--------------------------------------
Наследует tk.Text. Управляет атомарными метками изображений.

Внутреннее состояние:
  _images: dict[str, bytes]  # "pic1" → raw PNG/JPEG bytes

Метка [added picN]:
  - Вставляется как tk.Text tag с уникальным именем
  - Тег настроен на захват событий клавиш: любой KeyPress на символ
    внутри тега удаляет весь диапазон тега и соответствующий ключ из _images
  - При сборке сообщения для API метки заменяются на реальные данные
    изображения в формате base64

Сборка сообщения (get_message()):
  1. Обходим содержимое поля сегмент за сегментом
  2. Текстовые сегменты → content type "text"
  3. Теги изображений → content type "image_url" с base64 data URL
  4. Возвращаем список content-блоков для OpenRouter API


МОДУЛЬ: GeometryPhotoSolver (app/geometry_solver.py)
----------------------------------------------------
  endpoint: https://openrouter.ai/api/v1/chat/completions
  auth:     Bearer OPENROUTER_API_KEY
  format:   OpenAI-compatible JSON

  Конвейер solve_content_blocks():
    - извлекает text/image content blocks
    - при изображении строит full/text_crop/diagram_crop варианты
    - отправляет задачу в Qwen → Kimi → Llama
    - при недоступности parser/verifier уходит в degraded fallback
    - при необходимости делает self-check round
    - наружу возвращает только answer-only строку вида "1) 69"


МОДУЛЬ: Session (app/session.py)
---------------------------------
  Хранит историю текущей сессии в памяти:
    messages: list[dict]  # {role, content}

  История передаётся в каждый запрос к API (Сценарий 1).
  Сценарий 2 — отдельный одноразовый запрос без истории.
  При закрытии приложения история не сохраняется (in-memory only).


МОДУЛЬ: HotkeyManager (app/hotkeys.py)
----------------------------------------
  Использует pynput.keyboard.GlobalHotKeys.
  Регистрирует хоткеи из .env:
    HOTKEY_WINDOW    → toggle ChatWindow (через root.after для thread safety)
    HOTKEY_CLIPBOARD → запустить Scenario2Handler в отдельном потоке

  Все взаимодействия с tkinter выполняются через root.after() —
  прямые вызовы из потока pynput небезопасны.


ФОНОВЫЙ РЕЖИМ И WATCHDOG
--------------------------

run.sh:
  1. Проверяет наличие python3, pip
  2. Устанавливает зависимости из requirements.txt (если не установлены)
  3. Запускает: python3 main.py >> mcko.log 2>&1 &

main.py:
  - Создаёт tk.Tk() с withdraw() — окно скрыто с самого старта
  - Устанавливает wm_overrideredirect(True) чтобы убрать из панели задач
  - Запускает watchdog-поток

Watchdog (поток-демон):
  - Контролирует основной поток через событие-флаг
  - Если основной процесс завершился аварийно — перезапускает через subprocess
  - Между попытками перезапуска — пауза 3 секунды
  - Логирует каждый перезапуск в mcko.log

УСТАНОВКА НА ЧИСТУЮ ROSАOS
-----------------------------

run.sh выполняет:
  1. which python3 || (sudo apt-get install -y python3 python3-pip)
  2. pip3 install -r requirements.txt --quiet
  3. cp .env.example .env (если .env не существует)
  4. python3 main.py >> mcko.log 2>&1 &
  5. echo "MCKO запущен (PID: $!)"

Пользователь должен вручную заполнить .env после первого запуска,
если он был создан из шаблона.


ПОТОК ДАННЫХ: ИЗОБРАЖЕНИЕ В ЗАПРОСЕ
--------------------------------------

Сценарий 1:
  Clipboard → PIL.Image → BytesIO → base64 encode
  → сохранить в InputField._images["picN"]
  → при отправке: собрать content-блок {"type":"image_url","image_url":{"url":"data:image/png;base64,..."}}
  → передать в GeometryPhotoSolver.solve_content_blocks() вместе с текстом

Сценарий 2:
  Clipboard → PIL.ImageGrab.grabclipboard()
  → если Image: конвертировать в base64 PNG
  → если str: передать как текст
  → GeometryPhotoSolver.solve_content_blocks() → answer-only текст → Clipboard.write(text)


ЛОГИРОВАНИЕ
-----------
  Файл: mcko.log (рядом с main.py)
  Уровень: INFO (ошибки — ERROR)
  Формат: %(asctime)s [%(levelname)s] %(name)s: %(message)s
  Ротация: нет (для простоты; при необходимости — RotatingFileHandler)

  Логируются:
    - Запуск и остановка приложения
    - Каждый запрос к API (модель, кол-во токенов в запросе)
    - Ошибки API (статус-код, тело ответа)
    - Перезапуски watchdog
    - Ошибки хоткеев
