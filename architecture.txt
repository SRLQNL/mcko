  MCKO - CURRENT ARCHITECTURE
================================================================================

OVERVIEW
--------
MCKO is a small Python X11/Tkinter desktop app. It starts hidden, listens for
global hotkeys, accepts text/images/screenshots, sends tasks to OpenRouter, and
returns answer-only text.

The active solver runtime is `app/geometry_solver.py`.

Active model roles:
  Qwen2.5-VL-72B-Instruct  parser / OCR / visual extraction / option arbiter
  Kimi K2.6                primary solver
  Llama 4 Maverick         verifier / JSON repair model

The old single-model `APIClient` path is not active and `app/api_client.py` is
not part of the current runtime.

RUNTIME DATA FLOW
-----------------

GUI path:

  InputField content blocks
      -> main.on_send()
      -> Session.add_user()
      -> GeometryPhotoSolver.solve_content_blocks()
      -> ChatView answer-only output

Clipboard path:

  Ctrl+Alt+Space
      -> app.scenario2.handle_clipboard_hotkey()
      -> app.clipboard detects text/image
      -> GeometryPhotoSolver.solve_content_blocks()
      -> clipboard.write_text(answer)

Screenshot path:

  Ctrl+Shift+S
      -> app.screenshot.take_screenshot()
      -> ChatWindow.insert_image_bytes()
      -> InputField image label
      -> user sends normally


SOLVER PIPELINE
---------------

`GeometryPhotoSolver.solve_content_blocks()`:

  1. Extracts text and image blocks.
  2. Text-only requests use `_solve_text_only()` through Kimi.
  3. Image/image+text requests:
     - `_prepare_variants()` decodes image data URLs and may create crops.
     - `_call_qwen()` parses/OCRs the source.
     - `_call_kimi()` solves from source images plus compact Qwen parse.
     - `_call_llama()` verifies Kimi output.
     - `_compare_results()` scores consistency.
     - `_should_run_qwen_solver_challenge()` may run Qwen as an option arbiter
       for detected "write selected option numbers" tasks.
     - `_should_run_self_check()` may trigger a second Qwen/Kimi/Llama round.
     - `_pick_user_answer()` chooses the final answer.
  4. Recoverable provider failures return degraded fallbacks.
  5. Unexpected top-level errors return `1) Не удалось определить ответ`.

OpenRouter endpoint:

  https://openrouter.ai/api/v1/chat/completions

Request style:

  stream: false
  temperature: 0
  response_format: {"type": "json_object"}


JSON AND REPAIR
---------------

Models are requested to return strict JSON, but live providers can return prose,
empty content, malformed JSON, or unexpected shapes.

Current handling:

  - `_request_json()` wraps recoverable HTTP/network/parse failures.
  - `_extract_json_object()` extracts JSON from wrapped or noisy text.
  - Kimi malformed output is repaired through Llama where possible.
  - Local answer salvage is used only when it looks safe.
  - Option-style prose is repaired remotely before local salvage because
    numbered explanations can be mistaken for final answers.
  - `_normalize_result()` tolerates dict/list/str/scalar variants in repaired
    payloads.


ANSWER SELECTION
----------------

The selector prefers answer-only correctness over exposing uncertainty text.
It considers:

  - parser ambiguity
  - provider fallback state
  - solver repair state
  - independent verifier availability
  - answer agreement
  - answer confidence
  - whether the candidate looks like a final answer

For multiple-choice option-selection tasks, Qwen can be used as a focused
option arbiter only when the parser identifies the task format and model answers
disagree.

No hardcoded task-answer exact engine should be active.


UI AND THREADING
----------------

Tkinter must run on the main thread. Hotkey callbacks dispatch work through
`root.after()` where they touch UI.

Important UI constraints:

  - app starts hidden
  - chat window is small
  - close button hides, not destroys
  - `overrideredirect(True)` behavior is intentional
  - pasted images become `[added picN]` labels
  - screenshot insertion should not show the window unexpectedly
  - input is re-enabled after response worker completion


CONFIGURATION
-------------

Required:

  OPENROUTER_API_KEY

Model variables:

  PHOTO_SOLVER_KIMI_MODEL
  PHOTO_SOLVER_QWEN_MODEL
  PHOTO_SOLVER_LLAMA_MODEL

Hotkeys:

  HOTKEY_WINDOW
  HOTKEY_SHOW
  HOTKEY_CLIPBOARD
  HOTKEY_SCREENSHOT

`OPENROUTER_API_KEY` can be stored raw or base64-encoded. `app/config.py`
decodes base64 values that do not start with `sk-`.

Legacy variables are not active:

  OPENROUTER_MODEL
  OPENROUTER_MODELS
  SYSTEM_PROMPT_1
  SYSTEM_PROMPT_2


TESTING
-------

Local:

  python3 -m py_compile main.py app/*.py ui/*.py tests/*.py
  python3 -m unittest discover -s tests -p 'test_*.py'

Live model suite:

  python3 tests/run_solver_fixture_suite.py user-regression

The live suite requires network and OpenRouter quota. It is model-real and
image-only, but it is not a full GUI/X11 test.


KNOWN RISKS
-----------

  - Qwen upstream `429` can degrade image parse quality.
  - Kimi can return prose or malformed JSON.
  - Llama can confidently verify a wrong answer.
  - Option-selection geometry is the most unstable observed format.
  - The GUI needs target-machine X11 testing; local unit tests do not cover all
    focus/hotkey behavior.
