# MCKO — Фоновый AI-ассистент

## Описание проекта

Кроссплатформенное десктопное приложение на Python, работающее в фоновом режиме. Предоставляет два сценария взаимодействия с языковыми моделями через OpenRouter API: через всплывающее окно чата и через буфер обмена.

## Целевая платформа

- **ОС**: RosaOS (Linux-based), чистая установка без дополнительного ПО
- **Устройство**: ноутбук
- **Учётная запись**: admin с правами администратора
- **Пакетный менеджер**: dnf (основной), apt-get (тоже работает) — `run.sh` автодетектирует
- **Интернет на целевых устройствах**: недоступен — все pip-зависимости поставляются офлайн из папки `packages/`

## Установка и запуск

```bash
git clone <repo-url>
cd mcko
bash run.sh
```

Все зависимости устанавливаются автоматически при первом запуске. Ручная установка стороннего ПО не требуется.

## Офлайн-зависимости (pip)

Python-пакеты поставляются в папке `packages/` в виде wheel-файлов. PyPI не используется.

**Целевое окружение**: Python 3.8.13 (RosaOS), архитектура `x86_64`.

**Скачать wheels на машине с интернетом** (один раз, при обновлении зависимостей):
```bash
# Основные пакеты (бинарные wheels для Python 3.8 / manylinux)
pip3 download -r requirements.txt -d ./packages \
  --platform manylinux_2_17_x86_64 \
  --python-version 3.8 \
  --only-binary=:all:

# evdev (нет бинарного wheel — скачиваем sdist, компилируется на целевой машине)
pip3 download "evdev>=1.3,<1.9" -d ./packages --no-binary=evdev --no-deps

# python-xlib и six (зависимости pynput на Linux)
pip3 download "python-xlib>=0.17" six -d ./packages \
  --platform manylinux_2_17_x86_64 --python-version 3.8 --only-binary=:all:

# setuptools и wheel (нужны для сборки evdev из исходников)
pip3 download "setuptools<76" wheel -d ./packages \
  --platform manylinux_2_17_x86_64 --python-version 3.8 --only-binary=:all:
```

> **Важно**: `setuptools>=77.0` требует Python 3.9+. Используем `setuptools<76` (75.3.4).
> `evdev>=1.9` также требует setuptools>=77 — используем `evdev<1.9` (1.8.0).
> `pynput` на Linux зависит от `evdev` и `python-xlib` — их нужно добавлять явно.

Папку `packages/` нужно коммитить в репозиторий. При установке `run.sh` использует флаги `--no-index --find-links=./packages` — сеть не нужна.

**Порядок установки в run.sh**: сначала `setuptools` и `wheel`, затем остальные зависимости — иначе сборка evdev из sdist падает.

**Системные пакеты** (`python3`, `tkinter`, `xclip`, `gcc`, `python3-devel`) устанавливаются через пакетный менеджер:
- tkinter: `python3-tkinter` (dnf) / `python3-tk` (apt)
- build tools для evdev: `gcc python3-devel kernel-headers` (dnf) / `gcc python3-dev linux-headers-generic` (apt)

## Конфигурация (.env)

`.env` хранится в корне репозитория и коммитится в git. API-ключ хранится в **base64** — GitHub secret scanning не распознаёт его и не отзывает.

`config.py` автоматически определяет формат: если значение не начинается с `sk-` — декодирует из base64.

**Как закодировать новый ключ:**
```bash
python3 -c "import base64; print(base64.b64encode(b'sk-or-v1-...').decode())"
```

| Переменная | Описание |
|---|---|
| `OPENROUTER_API_KEY` | API-ключ (в base64) |
| `OPENROUTER_MODEL` | Идентификатор модели (например, `google/gemini-2.5-flash-lite`) |
| `SYSTEM_PROMPT_1` | Системный промпт для Сценария 1 (окно чата) |
| `SYSTEM_PROMPT_2` | Системный промпт для Сценария 2 (буфер обмена) |
| `HOTKEY_WINDOW` | Хоткей вызова окна (по умолчанию: `<ctrl>+<space>`) |
| `HOTKEY_CLIPBOARD` | Хоткей фонового запроса (по умолчанию: `<ctrl>+<alt>+<space>`) |

## Сценарии использования

### Сценарий 1 — Окно чата (Ctrl+Space)

- Вызов/скрытие окна глобальным хоткеем
- Многострочный ввод: `Shift+Enter` — перенос строки, `Enter` — отправка
- Вставка изображений из буфера через `Ctrl+V`:
  - Изображение не рендерится в поле ввода
  - Отображается атомарная метка `[added pic1]`, `[added pic2]` и т.д.
  - Удаление любого символа метки удаляет её целиком вместе с данными изображения
- История чата сохраняется в течение всей сессии (неограниченное число сообщений)
- Ответ ИИ отображается прямо в окне в виде чата

### Сценарий 2 — Фоновый запрос (Ctrl+Alt+Space)

- Работает всегда, независимо от состояния окна
- Пользователь копирует текст или изображение в буфер обмена
- Нажимает хоткей — программа забирает содержимое буфера и отправляет запрос
- Ответ записывается обратно в буфер обмена
- Никакое окно не открывается

## UI — окно чата

- **Стартовый размер**: 300×150, позиция — левый нижний угол экрана
- **Тип окна (X11)**: `overrideredirect(True)` — WM-декорации убраны. Кастомный тулбар (MCKO, ×) — закрытие и перетаскивание. Grip ◢ в правом нижнем углу — ресайз
- **Прозрачность**: `win.attributes("-alpha", 0.88)`
- **Поле ввода**: всегда видимо, пакуется `side=BOTTOM` до chat_frame — при любом размере окна не уходит за край
- **Высота поля ввода**: 2 строки (фиксированная), не конкурирует за пространство с чатом

## Фоновый режим

- Приложение запускается в фоне, не отображается в панели задач
- Нет видимых окон при запуске
- Незаметность для стороннего наблюдателя

## Обработка ошибок

- Логирование ошибок в файл `mcko.log`
- Автоматический перезапуск при аварийном завершении
- Watchdog-процесс контролирует основной процесс

## Исправления и текущие особенности

### X11 / RosaOS

- Окно использует `overrideredirect(True)` — убирает KDE-полоску. Фокус форсируется через Xlib `XSetInputFocus` (минует WM-политику, работает с overrideredirect). Ресайз — кастомный grip ◢.
- `_MOTIF_WM_HINTS` не убирает тайтлбар на KDE/RosaOS — не использовать.
- Kастомный тулбар (MCKO, ×) обеспечивает закрытие и перетаскивание.
- После показа окна нужно явно выполнять `deiconify()`, `lift()`, `focus_force()` и затем ставить фокус на `InputField` через `after_idle` / `after(...)`.

### Хоткеи

- Комбинации `Ctrl+Space` и `Ctrl+Shift+Space` пересекаются в `pynput.GlobalHotKeys`: на RosaOS при нажатии `Ctrl+Space` прилетают оба события (WINDOW и SHOW).
- Логика: `on_show_hotkey` отменяет pending toggle **только если окно скрыто** — тогда SHOW сам его покажет. Если окно видимо — toggle выполняется и скрывает его.
- Toggle выполняется с задержкой 120ms через `root.after()`, чтобы SHOW успел отменить его если нужно.

### OpenRouter / .env

- `.env` должен загружаться через `load_dotenv(..., override=True)`, иначе старые переменные окружения, особенно при запуске через `sudo`, могут перебить актуальный `OPENROUTER_API_KEY` из файла.
- `OPENROUTER_API_KEY` хранится в base64 в `.env` — защита от GitHub secret scanning (иначе OpenRouter отзывает ключ автоматически). `config.py` декодирует при старте.
- `OPENROUTER_API_KEY` нормализуется: убираются пробелы, BOM и внешние кавычки.
- Ошибка OpenRouter `401 {"message":"User not found."}` означает невалидный/отозванный ключ или проблему авторизации на стороне OpenRouter.

### Сервисные скрипты

- `kill.sh` — завершает все процессы проекта по шаблонам `main.py` и `watchdog.sh`.

## Правила разработки

- **После каждого изменения кода — коммит и push** в `main`. Целевая машина обновляется через `git pull`.

- **Логирование обязательно**: каждая функция, метод или значимый блок кода должны содержать вызовы `logger.info()` / `logger.debug()` / `logger.error()` в ключевых точках — запуск, завершение, входные параметры, результат, исключения. Логи пишутся в `mcko.log` через стандартный `logging` модуль. Не оставлять "немых" функций без единого лога.

- **Совместимость с Python 3.8**: целевая машина использует Python 3.8.13. Запрещено использовать:
  - `list[dict]`, `dict[str, int]` и другие дженерики встроенных типов в аннотациях — только через `typing` (`List`, `Dict`, `Tuple`) или `from __future__ import annotations` в начале файла
  - Оператор `|` для объединения типов (`X | Y`) — использовать `Union[X, Y]` из `typing`
  - `match`/`case` (доступно с 3.10)
  - `str.removeprefix` / `str.removesuffix` (доступно с 3.9)

## API

- Провайдер: [OpenRouter](https://openrouter.ai) (совместим с форматом OpenAI)
- Только один эндпоинт: `POST https://openrouter.ai/api/v1/chat/completions`
- Поддержка мультимодальных запросов (base64-изображения)
- История диалога передаётся в каждом запросе (в рамках сессии)
